Testing

Autograded assignments are subject to the submission limits given in the deadline policy and will be checked by several programs I wrote to test student code. These tests will run your program or its functions with different inputs and check whether it produces the correct results.

Academic Integrity and Accountability

Attempting to deliberately fool the test programs, hide malicious code in your source file, or otherwise intentionally sabotage tests is not allowed and will be treated as an AIA violation.

Test Criteria

Tests will normally check the following:

  • Output produced by IO.println and IO.print. Your program's output must match any requirements given in the assignment. Any difference (wrong spelling, capitalization, spacing, etc.) will fail an I/O test.
  • Values returned by your function. If your function returns a value, then the value must match any requirements given in the assignment.
  • Modifications to array arguments passed to your function. If your function takes any array parameters, then it must modify them as described in the assignment's requirements. Any extra modifications will cause a test to fail.

Your programs should avoid printing trailing whitespace (such as spaces at the end of a line) and should end their output with a println rather than a print, but the autograder (as of 11/14) will take the following steps to normalize your output:

  • trim trailing whitespace
  • delete any blank lines or lines that only contain whitespace
  • add a newline to the end of your output if one is missing

This means trailing whitespace, blank lines, and whether or not your output ends in a newline should not cause your program to fail any I/O tests. Spaces in the middle of a line or at the beginning of a line will still affect the outcome of a test.

Tests will always ignore any output produced by IO.readln. I never specify how exactly to prompt the user for input, so you may word these prompts however you want. The tests will only look at output from IO.print and IO.println. This also means you shouldn't use IO.readln to print any of the required output.

A few other ways for a test to fail:

  • Prompting for additional input or prompting in the wrong order. The test provides input in the order described in the assignment, so your code needs to read it in the same order.
  • Printing output in the wrong order, combining separate lines into the same line, or splitting up one line into several lines.

Random Numbers

If your program needs to generate random numbers, then it must use Math.random() to generate the number and scale, adjust, and round this value as needed. For example, (int)(Math.random() * 6) + 1 generates a random integer from 1 to 6.

Your program must also avoid generating unnecessary random numbers. The tests will seed the random number generator to ensure your code receives the same sequence of "random" values that were used when generating the expected output, and generating extra random numbers in your code will change this sequence and cause you to fail tests.

Test Feedback

My test program will generate markdown, html, and pdf files with the results of any failed tests. This feedback will provide some information about the failed test to help you find and fix any mistakes in your program.

Code Restrictions

To ensure your code is compatible with my tests, there are a few restrictions you need to follow. Ignoring any of these restrictions will usually cause your submission to fail all tests (or be marked invalid before even reaching the tests), and in some cases it will flag your code for review as potentially plagiarized or AI-generated.

These restrictions aren't a problem if you are writing your code as shown in class and only write code that you understand. If you don't know what something means and we haven't covered it in class (like a static modifier), then there's no reason you'd write it into your program.

Compact Source

You program must be a compact source file, and your main function must be declared as void main(). You must not use static or any other modifiers with your functions and variables.

I/O Functions

Any I/O must use the IO functions we learned in class: IO.readln, IO.print, and IO.println. No System.out, Console, or Scanner.

However, these projects are meant to be completed with the limited tools we cover in class, and the standard library can trivialize some problems that you're meant to solve by hand. You're free to make use of any String methods, any of the IO and Math functions we've covered, and parsing functions (like Integer.parseInt). However, using other parts of the standard library that we haven't covered, such as System or Scanner, is not allowed. You're also not allowed to use any third-party libraries in autograded assignments.

Character Sheet Project

Dungeons and Dragons

Dungeons and Dragons is a tabletop roleplaying game (TTRPG). Even if you haven't played a TTRPG before, you've probably still heard the name "Dungeons and Dragons" at some point. These assignments will use a modified version of some character creation rules from D&D 5e. The real rules can be found in the 5e System Reference Document (SRD), which you can read if you're curious and want to learn more. However, you do not need to read the SRD for this assignment.

Character Sheet

The idea behind these assignments is to write a program that generates a "character sheet" for someone playing a TTRPG like D&D. A player normally controls a fictional character (their Player Character, or PC), and they track information about their character on a character sheet. This might include statistics, like how strong or fast the character is, the character's possessions, or information about the character's personality.

The character sheet you generate will not cover all of the information about a real TTRPG character. This would get too complicated for an early programming assignment. Instead, we'll use a small subset of rules from the SRD, some of which will be simplified to better fit this assignment. You don't need to know any of these rules! The assignments will explain any calculations your code needs to perform.

Terminology

Below is a summary of some terms that will come up in this assignment:

  • A character's class is the role they fill in the game's world. In a game like D&D, which focuses on fighting fantasy monsters, a character's class tells us their role in combat. A wizard casts spells, a rogue sneaks up on enemies, and a fighter... fights. Yes, everyone fights, but the fighter fights more directly, like with a sword and shield.
  • A character's ancestry or race (the term varies by game, but I'll usually default to ancestry) represents what kind of creature they are. D&D includes a lot of your standard fantasy races: elf, human, dwarf, goblin, gnome, halfling, and orc. However, depending on the game and who's running it, a TTRPG character could be anything: a ghost, a dragon, a talking bear, or a mannequin. I have run a game of D&D that included both a talking bear and a mannequin as player characters. The mannequin also talked, and his name was Bob. The bear had a pet monkey. All of this will be on the exam1.
  • A character's hit points (HP) represent how difficult they are to defeat in combat.
  • A character's armor class (AC) represents how hard they are to hit with an attack. A high AC may mean that a character is very good at avoiding attacks, or that a character is wearing armor that blocks most attacks.
  • A character's ability scores are numbers that represent a character's strength, cunning, speed, and other innate qualities. These affect most actions in the game.
  • A character's ability modifiers are bonuses or penalties derived from their ability scores. The modifiers are the numbers we actually use in most calculations, and the scores only exist to determine the modifiers. At this point, you may be wondering why we bother with ability scores if we don't use them after calculating the modifiers. Good question! Anyway, let's talk about the character's level.
  • A character's level represents how powerful and experienced they are. In most TTRPGs, characters gain experience and level up to represent how they've grown during their adventures.
  • In addition to standard six-sided dice (referred to as D6 dice), D&D uses several less common dice shapes: D4, D8, D10, D12, and D20.

Dice Game Project

This is some background information for assignments 5-8. You should familiarize yourself with the rules of Yahtzee before starting these assignments.

Yahtzee

Yahtzee is a game played with five dice per player over 13 rounds. Each player's goal is to earn the most points by the end of the game. A round of Yahtzee consists of the following steps:

  1. Each player rolls all of their dice.
  2. Each player chooses any number of their dice to reroll and rerolls all of the chosen dice at the same time. They can then repeat this process one more time.
  3. Each player chooses one of thirteen categories on their scorecard. Each category can only be chosen once per player and is worth points based on the dice rolled. For example, choosing the yahtzee category is worth 50 points if all of your dice show the same number and 0 points otherwise.

At the end of the game, each player adds up their points. The player with the most points wins.

Our Dice Game

We'll be coding a dice game with most of the same rules as Yahtzee, but we'll leave out or simplify a few rules. Our game will be single-player, with one person playing on the terminal by typing in commands and trying to earn the highest possible score.

Score Categories

The game allows the user to choose from ten different score categories:

  1. Aces
  2. Twos
  3. Threes
  4. Fours
  5. Fives
  6. Sixes
  7. 3 of a Kind
  8. 4 of a Kind
  9. Yahtzee
  10. Chance

The rules for calculating these scores are as follows:

  • The first six categories (the "upper categories" in Yahtzee) are scored the same as in previous assignments: the category number multiplied by how many dice equal that number.
  • The 3 and 4 of a kind categories follow similar rules:
    • If the user has at least three (3 of a kind) or at least four (4 of a kind) dice with the same number, then these categories are worth the sum of all dice (not just the matching dice).
    • If the user doesn't meet the 3/4 of a kind requirement, then these categories are worth 0 points.
  • The Yahtzee category is worth 50 points if all of the dice are the same and 0 points otherwise.
  • The chance category is always worth the sum of all dice.
